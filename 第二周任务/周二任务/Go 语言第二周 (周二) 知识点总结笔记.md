一、 Goroutine（协程）的“轻量级并发”
Goroutine 是 Go 语言的并发执行单元，比线程更轻量：

- 创建简单：只需在函数调用前加上 `go` 关键字
- 调度高效：由 Go 运行时（GMP 模型）管理，可在少量线程上调度大量协程
- 内存占用小：初始栈大小仅 2KB，可动态扩容

在代码中：`go func() { ... }()` 启动了一个匿名函数的协程执行。

二、 Channel（通道）的“数据管道”
Channel 是 Goroutine 之间通信的桥梁，遵循“不要通过共享内存来通信，而要通过通信来共享内存”：

- 有缓冲通道：`make(chan int, 5)` 创建容量为5的缓冲通道
- 无缓冲通道：`make(chan int)` 创建同步通道，发送和接收必须同时就绪
- 通道操作：
  - 发送：`ch <- value`（如果通道满，发送方会阻塞）
  - 接收：`value := <- ch`（如果通道空，接收方会阻塞）
  - 关闭：`close(ch)`（由发送方关闭，关闭后可接收剩余数据但不可发送）
  - 遍历：`for value := range ch`（自动接收直到通道关闭）

在代码中：`numbers` 通道用于传递数字，生产者发送 `numbers <- i`，消费者通过 `for num := range numbers` 接收。

三、 sync.WaitGroup 的“协程等待组”
WaitGroup 用于等待一组 Goroutine 完成工作：

- `wg.Add(1)`：增加等待计数（应在启动协程前调用）
- `wg.Done()`：减少等待计数（通常用 `defer wg.Done()` 确保执行）
- `wg.Wait()`：阻塞直到等待计数归零

在代码中：主协程使用 `wg.Wait()` 等待生产者和消费者协程完成，确保程序不会提前退出。

四、 生产者-消费者模式的“并发协作”
这是并发编程的经典模式：

- 生产者：生成数据并发送到通道
- 消费者：从通道接收并处理数据
- 缓冲通道：作为中间缓冲区，平衡生产速度和消费速度

在代码中：生产者生成1-15的数字，消费者计算平方并输出，缓冲通道容量5使得生产可稍微超前于消费。

五、 有缓冲通道的“流量控制”
缓冲通道的关键行为：

- 当通道未满时：发送立即完成，不阻塞
- 当通道已满时：发送操作阻塞，直到有空间
- 当通道为空时：接收操作阻塞，直到有数据
- 关闭通道后：接收方可继续接收剩余数据，之后 `range` 循环结束

在代码中：通道容量5意味着生产者最多可提前生产5个数字，然后必须等待消费者处理。

六、 并发编程的“安全退出模式”
确保并发程序正确退出的要点：

1. 发送方负责关闭通道（`close(numbers)`）
2. 接收方使用 `range` 自动检测通道关闭
3. 使用 `WaitGroup` 等待所有协程完成
4. 主协程最后调用 `wg.Wait()` 确保清理

在代码中：生产者完成所有发送后关闭通道，消费者 `range` 循环在通道关闭后自动退出，`wg.Wait()` 确保两者都完成。

七、 实践中的“性能与资源平衡”
- 缓冲大小选择：太大浪费内存，太小降低吞吐量
- 协程数量：根据任务类型和系统资源合理设置
- 错误处理：生产消费中应考虑错误传递机制

八、 今日避坑指南
1. **通道关闭原则**：永远由发送方关闭通道，重复关闭会导致 panic
2. **WaitGroup 使用顺序**：先 `Add()` 后启动协程，避免竞态条件
3. **缓冲通道的死锁**：如果所有协程都在等待通道操作，且没有其他协程可推进，会导致死锁
4. **资源泄漏**：不使用的协程如果不退出会一直占用资源
5. **数据竞争**：虽然通道可避免共享内存，但访问其他共享数据仍需同步机制（如 Mutex）
