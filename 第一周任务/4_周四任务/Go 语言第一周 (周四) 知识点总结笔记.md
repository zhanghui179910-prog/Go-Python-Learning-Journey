鸭子类型（Duck Typing）与面向接口编程

核心痛点：在 Python 中一切皆动态，但在 Go 这种静态强类型语言中，如何让主程序灵活调用不同类型的数据导出器？

通俗理解：面向接口编程的核心就是“制定岗位标准，不挑剔具体员工”。只要我的结构体（比如 JSONExporter）满足了接口（DataExporter）的标准（即拥有完全同名、同参数、同返回值的 Export 方法），就可以通过接口被统一调用。

避坑指南：Go 的鸭子类型是“隐式”的，不需要写 implements。只要方法签名对得上，系统自动认领。这带来了极大的方便和扩展性（热插拔）。

二、 空接口 any 与类型扫描仪 data.(type)

空接口（any / interface{}）：因为没有任何方法限制，所以它可以接收任何类型的数据，相当于 Python 里的 Any。

类型断言：当数据变成 any 传入时，编译器会“失忆”。data.(type) 是 Go 的专属内置关键字用法，它就是一个“类型扫描仪”。

使用规范：data.(type) 只能配合 switch 语句使用，用来拆开 any 的盲盒，匹配并还原数据的真实类型。

三、 抛弃 try-catch：Go 的 error 哲学

核心差异：Go 没有 Python 那种把异常抛到半空的 try-catch 机制。在 Go 里，error 就像 float64、string 一样，只是一个普通的内置数据类型。

规范操作：每次调用可能失败的函数（如 json.Marshal），都要紧跟一句 if err != nil。虽然写起来略显繁琐，但代码的执行流向极其清晰，逼迫开发者在错误发生的当下立刻处理。

四、 panic 与 recover：生死一线的保命机制

panic（恐慌）：这是保护程序的终极机制，同时也是内置方法。遇到绝对无法挽回的致命错误（如缺少核心配置、数组严重越界）时触发，直接掐断当前程序执行流，如果不拦截，整个程序将崩溃退出。

recover（恢复）：初学者常误以为它是用来“恢复丢失的数据”的，其实不是。recover 必须放在 defer 里，它的真正用处是：在遇到严重问题（panic）时，拦截崩溃过程，保证主程序（比如整个 Web 服务器或外层 for 循环）可以继续存活并运行下去。
